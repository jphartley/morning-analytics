# OpenSpec Queue Follow-Up Handoff

## Background

This document is the queue-architecture follow-up from the `add-welcome-empty-state` validation run. It supersedes the earlier first-run handoff by removing issues that did not recur and focusing on items that were either new or were supposed to be fixed by the prior queue hardening work but were not fixed thoroughly enough.

Use this document as context for a follow-up OpenSpec change against the Parallel OpenSpec Delivery Queue. It focuses only on queue workflow behavior: Builder worktree isolation, candidate setup, command environment, OpenSpec command noise, finalization, archive handling, and planning-checkout cleanup.

## Current Assessment

The previous `queue-architecture-hardening` change was meant to address several of these categories. The latest run shows that some mechanics exist in the queue script, but the actual assistant workflow can still bypass them or continue after a failed sub-step.

Do not re-open items that did not recur in this validation run:

- Worktree sandbox / writable-root permission noise.
- Dev server readiness reporting.
- The original landing-worktree failure caused by `main` already being checked out.
- Manual queue-state JSON recovery.

## Queue Architecture Issues

### Builder Worktree Boundary Regression

This was meant to be fixed by the prior queue hardening work, but implementation edits still landed in the planning checkout before being reapplied to the candidate worktree.

What happened in this run:

- After the candidate worktree existed, an implementation patch for app files was applied in `/Users/jhartley/code/morning-openspec`.
- The mistake was detected because the candidate diff was empty while the planning checkout had app changes.
- The agent manually reapplied the patch to the candidate worktree and removed only its accidental planning-checkout app edits.

Why the previous fix was incomplete:

- The queue has a Builder preflight concept, but the assistant workflow can still run an edit before that preflight is invoked or enforced.
- Absolute candidate paths in instructions are not sufficient if the active editing tool call still targets the planning checkout.

Desired outcome:

- Make `builder-preflight <change>` a required, visible step immediately before implementation edits.
- Require Builder instructions to include the candidate absolute path and a short "edit only under this path" boundary.
- Add or strengthen a post-implementation contamination check that fails loudly if the planning checkout received implementation, task, or candidate-only artifact edits.
- Consider adding wrapper guidance that tells assistants to use absolute file paths under the candidate worktree for every implementation patch after queue start.

### Candidate Bootstrap Still Too Easy To Bypass

This was also meant to be fixed by the prior queue hardening work. The queue script now has setup behavior, but the run still reached raw verification commands before the candidate environment was ready.

What happened in this run:

- The candidate worktree did not have `app/node_modules`.
- `npm run lint` failed with `eslint: command not found`.
- The agent manually ran `npm ci` in the candidate `app/` directory before rerunning lint/build.
- `npm ci` warned that the shell was using Node `v26.0.0` while the app pins Node `22.x`.
- The first build failed at prerender because `NEXT_PUBLIC_SUPABASE_URL` was missing.
- The build passed after rerunning with safe placeholder Supabase env values from the repo example.

Why the previous fix was incomplete:

- Queue-owned setup exists, but the workflow still allowed direct `npm run lint` / `npm run build` before setup.
- Setup and verification inherit the current shell environment; they do not force or select the repo-pinned Node 22 runtime.
- Placeholder env can be generated by queue setup, but a manually-run build path can still miss it.

Desired outcome:

- Make `setup <change>` or `prepare-test <change>` the required path before any lint/build/serve verification in a candidate.
- Have queue-managed commands run under the repo-pinned Node 22 runtime, or fail with a clear instruction before running `npm ci`, lint, build, or dev server commands.
- Report dependency state, Node version, and env mode together before verification starts.
- Avoid documenting raw `npm run lint` or `npm run build` in candidate handoffs unless they are shown as being run through the prepared candidate environment.

### OpenSpec Telemetry Noise

This appears to be new in this validation run.

What happened:

- `openspec status` and `openspec instructions apply` completed, but printed PostHog / `edge.openspec.dev` network errors.
- The command output was still usable, so the agent treated the telemetry errors as non-blocking.

Why it matters:

- Network/telemetry warnings make it harder to see whether an OpenSpec command itself failed.
- They add noise to queue logs and can make a normal step look suspicious.

Desired outcome:

- Suppress or disable OpenSpec telemetry in queue-managed commands when supported.
- If telemetry cannot be disabled, classify known telemetry/network failures separately from command failure in queue output.

### Archive Failure Did Not Stop Finalization

This was meant to be fixed by queue-owned finalization and archive flow work, but the failure mode was not handled correctly.

What happened in this run:

- `node scripts/openspec-queue.mjs finalize add-welcome-empty-state --confirm-gate2` ran.
- OpenSpec archive reported that the existing `openspec/specs/app-shell/spec.md` main spec is structurally invalid because requirements are outside a main `## Requirements` section.
- The archive step printed `Aborted. No files were changed.`
- Finalization still squash-merged and pushed `main`.
- The active `add-welcome-empty-state` change artifacts remained under `openspec/changes/` on `main`.

Why the previous fix was incomplete:

- The queue runs `openspec archive`, but it does not reliably treat a semantic archive abort as a finalization blocker.
- The finalization flow can leave `main` in a partial state: implementation merged and pushed, but OpenSpec artifacts unarchived.
- Output mixes an archive abort with a successful squash/push path, which makes the final state easy to misread.

Desired outcome:

- Add an archive preflight that validates touched main specs before merge/push finalization.
- If archive fails or reports no files changed because of a structural spec problem, stop before squash merge and push.
- If the queue intentionally supports partial finalization, require explicit recovery approval and label the result as partial.
- Make finalization output separate `archive`, `merge`, `push`, and `cleanup` statuses clearly.

### Planning Checkout Cleanup And Sync

This issue partially overlaps prior archive-flow cleanup work, but this run exposed two additional cleanup gaps.

What happened in this run:

- After finalization push, the planning checkout was still one commit behind `origin/main`.
- The planning checkout still had duplicate untracked pre-Gate OpenSpec files that had been created locally before queueing.
- There was also an unrelated pre-existing untracked file, `docs/openspec-queue-followup-handoff.md`, which the agent correctly preserved.
- The agent manually removed only the duplicate OpenSpec files it had created, then ran `git pull --ff-only` to sync local `main`.

Why the previous fix was incomplete:

- Prior archive-flow work addressed candidate archive state and duplicate active artifacts, but cleanup did not leave the planning checkout synchronized after a detached landing push.
- The workflow relies on the agent to distinguish queue-created untracked artifacts from unrelated user files.

Desired outcome:

- After a successful push from a landing worktree, report whether the planning checkout is behind `origin/main`.
- Offer or perform a safe fast-forward of the planning checkout when it is clean enough and when unrelated user files can be preserved.
- Track pre-Gate artifacts created by the workflow so duplicate cleanup is precise and does not depend on manual memory.
- Never delete unrelated untracked files; report them separately as preserved local state.
